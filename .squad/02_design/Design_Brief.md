## 2. 디자인해야 할 화면

### P0 (MVP 필수 — 반드시 포함)

| 화면 | 설명 |
|------|------|
| **메인 피드 페이지** | 뉴스 / VC 블로그 탭 구분, 아이템 목록 표시 |
| **피드 아이템 카드** | 출처, 플랫폼, 요약, 원본 링크, 발행 시각 포함 |
| **빈 피드 상태** | 수집 중 또는 데이터 없을 때의 상태 |
| **로딩 상태** | 피드 불러오는 중 스켈레톤 or 스피너 |
| **에러 상태** | 크롤링 실패 등 오류 시 표시 |

### P1 (MVP 포함 가능 — 여유 있으면)

| 화면 | 설명 |
|------|------|
| **키워드 필터 UI** | 검색창 or 칩 형태, URL 파라미터 연동 |
| **검색 결과 피드** | 키워드 검색 후 필터링된 아이템 목록 |

---

## 3. 각 피드 아이템에 표시되는 데이터

```
- 출처명 (인물명 또는 기관명)
- 플랫폼 태그 (예: "플래텀", "카카오벤처스 블로그")
- 제목 또는 요약 텍스트
- 발행 시각 (상대 시간: "3시간 전")
- 원본 링크 (외부 탭으로 열기)
```

---

## 4. 사용자 특성 (디자인 방향 힌트)

- **타겟**: VC 심사역, 스타트업 창업자 — 정보 밀도 높게, 빠른 스캔 가능해야 함
- **사용 맥락**: 매일 아침 훑어보기 → 인스타그램이 아닌 Hacker News/TechCrunch 느낌
- **디바이스**: 데스크톱 우선 (모바일 반응형은 보너스)
- **인터랙션**: 클릭하면 원본으로 이동이 전부 — 복잡한 인터랙션 없음

---

## 5. 디자인 레퍼런스 방향

- Hacker News, TechCrunch, Feedly 같은 뉴스 집약 서비스 스타일
- 깔끔하고 텍스트 중심 — 화려한 UI보다 정보 밀도
- 다크모드 불필요 (라이트모드만)

---

## 6. 기술 제약 (Designer가 알아야 할 것)

- Tailwind CSS 사용 — 디자인 토큰을 Tailwind 클래스 기준으로 정의
- Next.js App Router — 페이지 라우팅은 `/` (메인), `/search` (검색) 수준
- 애니메이션 최소화 — 빠른 렌더링 우선
